Install · Hermes

Install Nio for Hermes Agent.

Two surfaces, one setup.sh: shell-hooks for the guard + observability pipeline, and a small Python plugin for the /nio slash command. Bypass the LLM entirely on both.

Requirements

  • Hermes Agent installed and hermes on PATH
  • Node.js ≥ 18 on PATH (the bundled CLIs are Node)
  • macOS, Linux, or WSL
Shell-hook support

The shell-hook subsystem Nio plugs into landed via upstream PR #13296. If your Hermes is older, hermes hooks --help will fail or omit the list / doctor / test subcommands. Upgrade Hermes before running setup.sh.

Install

One-liner

VERSION=$(curl -s https://api.github.com/repos/core0-io/nio/releases/latest | grep tag_name | cut -d'"' -f4) && \
curl -LO "https://github.com/core0-io/nio/releases/download/${VERSION}/nio-hermes-${VERSION}.zip" && \
unzip -o "nio-hermes-${VERSION}.zip" -d nio-hermes && \
cd nio-hermes && ./setup.sh

Manual

  1. Grab nio-hermes-v<version>.zip from the Releases page.
  2. unzip nio-hermes-v<version>.zip -d nio-hermes
  3. cd nio-hermes && ./setup.sh

If you also use Claude Code or OpenClaw, grab nio-all-v<version>.zip instead — the bundled setup.sh auto-detects which agents you have and installs Nio for each.

Two surfaces

Hermes integration installs along two orthogonal channels:

ChannelWhatHow
Shell-hooks Hot path: guard (Phase 0–6) + OTEL collector (metrics, traces, logs). Seven entries merged into ~/.hermes/config.yaml; Hermes spawns hook-cli.js on each event.
Python plugin User-facing /nio slash command — bypasses the LLM, mirrors OpenClaw's command-dispatch: tool. Files dropped into ~/.hermes/plugins/nio/; Hermes auto-discovers and registers /nio on startup.

Shell-hook events

Same command string across all seven, so Hermes's allowlist only needs approving once:

EventPipeline branch
pre_tool_callGuard — Phase 0–6, can deny / confirm / allow
post_tool_callCollector — closes tool span, emits nio.tool_use.count
pre_llm_callCollector — opens turn span, captures user prompt
post_llm_callCollector — closes turn span, emits nio.turn.count
on_session_startCollector — session boundary marker
on_session_endCollector — defensive turn close + flush
subagent_stopCollector — subagent turn close

What setup.sh does

  1. Verifies Node.js ≥ 18 is on PATH.
  2. Merges the seven event entries into ~/.hermes/config.yaml with a timestamped .bak backup.
  3. Installs the /nio Python plugin into ~/.hermes/plugins/nio/ and appends "nio" to plugins.enabled (Hermes plugins are opt-in).
  4. Optionally pre-approves the hooks for non-TTY runs — see Consent below.
  5. Creates ~/.nio/ and writes config.yaml from defaults (kept if it already exists, unless --reset-config).

Consent

Hermes refuses to fire unknown shell hooks until you consent. Approval is persisted to ~/.hermes/shell-hooks-allowlist.json, keyed on the exact command string. Four ways to approve:

PathHowScope
Interactive (TTY) setup.sh ends with Approve this hook now? [y/N] — answer y. This exact command string only.
Non-interactive bash plugins/hermes/setup.sh --accept-hooks (or ./setup.sh --accept-hermes-hook at the top level). Same — scoped to this command.
Hermes-side, on next run hermes chat --accept-hooks — type exit to leave. Same — scoped to declared shell hooks.
Blanket HERMES_ACCEPT_HOOKS=1 env var, or hooks_auto_accept: true in ~/.hermes/config.yaml. All shell hooks globally — not just Nio.
Why hermes hooks doctor --accept-hooks doesn't work

hermes hooks doctor and hermes hooks test don't write to the allowlist — --accept-hooks is silently a no-op on those subcommands. Approval only persists when an agent path (chat, acp, rl) starts up. setup.sh --accept-hooks handles this for you without spawning a chat session, so you don't need to hermes chat manually.

Flags

FlagPurpose
--accept-hooksApprove hooks non-interactively (no TTY prompt). Required for CI / cron / gateway-only installs.
--dry-runPrint the resulting config.yaml without writing.
--yesSkip interactive merge prompts (CI / scripting).
--uninstallRemove the seven hook entries, the Python plugin directory, and the allowlist entry.
HERMES_CONFIG_PATH envOverride target config path (default ~/.hermes/config.yaml).

Where things live

PathWhat
~/.hermes/config.yamlHermes runtime config — Nio appends a hooks: block + "nio" to plugins.enabled.
~/.hermes/shell-hooks-allowlist.jsonApproved hook command strings (Hermes-managed).
~/.hermes/plugins/nio/Python plugin: plugin.yaml, __init__.py, scripts/{nio-cli,hook-cli}.js.
~/.nio/config.yamlNio's guard + collector config (shared with Claude Code / OpenClaw). See Config reference.
~/.nio/audit.jsonlAppend-only JSONL audit log: guard decisions, scans, lifecycle events, and every hook event (pre_tool_call, post_tool_call, on_session_start, on_session_end, …). Read via /nio report.
~/.nio/traces-state-store.jsonCross-process trace state (turn IDs + pending span starts). Internal; sits next to audit.jsonl.

Set NIO_HOME to move the Nio config/audit log elsewhere, or set collector.logs.path in config.yaml to relocate just the audit log + state file. Hermes's own HERMES_CONFIG_PATH override is independent.

Verify

  1. hermes hooks doctor — expect "Checking 7 configured shell hook(s)…", all green, JSON smoke test passes for each.
  2. If a Hermes gateway is already running, restart so it loads the new config + allowlist: hermes gateway run --replace.
  3. Trigger a tool call (chat, Telegram, Discord — any channel). Watch ~/.nio/audit.jsonl for a platform: "hermes" guard entry.
  4. In any Hermes channel, send /nio config show — bot replies with the current Nio config JSON. No LLM in the loop.
If /nio isn't responding

Hermes plugins are opt-in. Confirm:

grep -A3 '^plugins:' ~/.hermes/config.yaml | grep nio   # should print "- nio"

If nio is missing, re-run ./setup.sh — the script appends it idempotently. After any change, restart any running gateway with hermes gateway run --replace.

Re-approval after upgrade or move

The allowlist entry is keyed on the exact command string in config.yaml. Anything that changes the absolute path of hook-cli.js — moving the install directory, upgrading to a new release zip, switching machines — requires re-approval. setup.sh --accept-hooks handles this automatically: it revokes the stale entry first, then re-registers with the new path.

Upgrade

Download the new zip, run ./setup.sh --accept-hooks. The seven hook entries are merged per-event idempotently. Restart any running gateway to pick up new bindings: hermes gateway run --replace.

Uninstall

./setup.sh --uninstall

Removes the seven hook entries from ~/.hermes/config.yaml, deletes ~/.hermes/plugins/nio/, and revokes the allowlist entry. Safe to re-run.
