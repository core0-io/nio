Tool Gate — is this tool blocked, available, or guarded?

The first checkpoint. Looks at the tool name only — no command, no file content. Decides whether to deny outright (blocked), pass through without further analysis (available, not guarded), or hand the call to Phase 1+ (guarded).

PropertyValue
Latency<1ms
ScopeAll tools
TypeGate (no score emitted)
Weight—
Short-circuitblocked → deny · available (not guarded) → allow

How the gate decides

  1. Look up the tool name in guard.blocked_tools for the active platform (and mcp if it's an MCP tool). Match → deny.
  2. If guard.available_tools for the platform is non-empty AND the tool isn't on it → deny (whitelist mode).
  3. Look up the tool in guard.guarded_tools. Found → continue to Phase 1 with the mapped action type. Not found → allow (auto-approve, log only).

Rules

RuleDescription
blocked_toolsHard-deny list — never runs.
available_toolsKnown tools, skip the rest of the guard.
guarded_toolsHand off to Phase 1+ deep analysis.

Configuration that affects this phase

  • guard.blocked_tools — per-platform + mcp denylist.
  • guard.available_tools — per-platform + mcp allowlist (whitelist mode when non-empty).
  • guard.guarded_tools — per-platform tool → action type mapping.

MCP nuance

The mcp key is platform-agnostic and applies whenever the incoming tool is an MCP tool, including mcporter-style shell invocations like npx mcporter call hass.HassTurnOn. Entries accept either a bare local name (HassTurnOn) or server-qualified (hass__HassTurnOn). Matching is case-insensitive.

Examples

guard:
  blocked_tools:
    claude_code: ['WebSearch', 'WebFetch']
  available_tools:
    claude_code: []         # empty → no whitelist mode
    mcp: ['hass__*']        # whitelist of MCP tools (any platform)
  guarded_tools:
    claude_code:
      Bash: exec_command
      Write: write_file
      Edit: write_file
