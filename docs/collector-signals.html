Collector Signals — what Nio captures while an agent runs

Nio emits three OTEL signals — metrics, traces, and logs — covering every tool invocation, every guard decision, every conversation turn. Schemas are aligned around shared GenAI semantic-convention keys (gen_ai.tool.name, gen_ai.conversation.id, nio.guard.*) so the same dashboards work across all three signals. The audit log is the only signal with a local JSONL backup; metrics and traces are OTLP-only.

Pipeline overview

The three platform adapters (Claude Code, Hermes, OpenClaw) all funnel into a single dispatcher that fans out to the three OTEL signals. CC and Hermes share the spawn-per-hook entry; OpenClaw is in-process and bypasses dispatchCollectorEvent, calling the trace pure functions directly.

Metrics

Schema declared in the METRICS_SCHEMA constant. Four instruments. Label keys are aligned with the matching trace span attributes (gen_ai.tool.name matches the tool span; nio.guard.* matches the OpenClaw tool-span guard attrs via nioGuardAttributes) so cross-signal queries (metrics ⇌ traces ⇌ logs) use the same keys.

Claude Code · Task → Agent

The user-facing Task tool (subagent dispatch) is reported as tool_name="Agent" in CC hook payloads, so PreToolUse / PostToolUse counters use Agent as the gen_ai.tool.name label. The literal value Task only appears as a counter label when TaskCreated / TaskCompleted fire (Teammates / cloud-agent flows; never fired by the regular Task tool subagent on current CC builds — see e2e-test/hook-subagent-e2e-task.md). OpenClaw and Hermes use their own native tool names.

Metrics have no local file — there is no metrics.jsonl. If collector.endpoint is empty, metrics drop on the floor (the meter provider returns null).

Traces

One trace per conversation turn. Span hierarchy follows the OTel GenAI semantic conventions where applicable; Nio-specific extensions use the nio.* prefix. Schema lives in traces-collector.ts — every literal attribute key string is owned by one helper there; consumers (collector-core.ts, openclaw-plugin.ts) only call helpers.

Trace: invoke_agent UserPromptSubmit  (root, opens at 1st PreToolUse, ends at Stop / SubagentStop)
  ├─ Span: execute_tool <name>   (PreToolUse → PostToolUse)
  ├─ Span: execute_tool <name>   (...)
  └─ Span: task:execute          (TaskCreated → TaskCompleted, or OpenClaw subagent_spawning → subagent_ended)

Trace state and span lifecycle

All three platforms route span construction through traces-collector.ts pure functions (ensureTurn, recordPreToolUse, recordPostToolUse, recordPreTaskToolUse, recordPostTaskToolUse, setTurnAttributes, endTurn). The only platform difference is where CollectorState lives:

  • Claude Code / Hermes — cross-process spawn-per-hook; bridged via traces-state-store.json on disk. PreToolUse writes pending span data; PostToolUse reads it and emits the span retroactively with the original start time. Stop (or SubagentStop / SessionEnd) emits the turn root span.
  • OpenClaw — in-process daemon; per-session Map<sessionId, CollectorState> in memory inside openclaw-plugin.ts. No disk bridging; same lifecycle methods.

State file path: dirname(collector.logs.path) / traces-state-store.json (defaults to ${NIO_HOME ?? ~/.nio}/traces-state-store.json).

Logs (audit log)

Audit entries are dual-written: OTEL Logs export to <endpoint>/v1/logs (when collector.logs.enabled) AND a local JSONL file at collector.logs.path (when collector.logs.local). The JSONL line is the entry verbatim; the OTEL LogRecord uses body = JSON.stringify(entry) plus a flat attribute list for indexing, built by auditEntryAttributes. Schema is the discriminated union AuditEntry.

Configuration

All three signals are gated by the collector section in ~/.nio/config.yaml. Full field-by-field reference: Configuration · collector.

YAML cheatsheet (click to expand) 
collector:
  endpoint: ""                      # OTLP base URL; empty = no OTLP export at all
  api_key: ""                       # Bearer token
  timeout: 5000                     # milliseconds
  protocol: http                    # http | grpc
  metrics:
    enabled: true                   # OTLP metrics export on/off
  traces:
    enabled: true                   # OTLP traces export on/off
  logs:
    enabled: true                   # OTLP logs export on/off
    local: true                     # local JSONL backup on/off
    path: "~/.nio/audit.jsonl"      # audit log + (sibling) traces-state-store.json
    max_size_mb: 100                # rotation threshold for the local file
  • When collector.endpoint is empty, the corresponding provider factory returns null and the platform code skips emit.
  • The audit-log local JSONL still works (controlled by collector.logs.local) even without an endpoint — handy for offline / air-gapped use.

Source-of-truth files

When schema drifts, update the source file and re-run pnpm run build. The helper functions in traces-collector.ts are the single point of change for trace attribute keys — most edits don't ripple beyond one file.

SchemaFileWhat it owns
Metrics catalog metrics-collector.ts (METRICS_SCHEMA) Instrument names, types, labels, descriptions
Trace span attributes traces-collector.ts All gen_ai.* and nio.* span attribute keys via builder helpers (genAi*Attributes, nio*Attributes, record* operations)
Trace state shape traces-state-store.ts CollectorState, PendingToolSpan, PendingTaskSpan
Audit log entry types audit-types.ts All entry shapes (guard / scan / lifecycle / config_error / hook)
OTEL log attribute projection logs-collector.ts (auditEntryAttributes) Flat attribute keys for log indexing — shared GenAI / nio.* keys with the trace signal
Collector config schema config-schema.ts CollectorConfigSchema, CollectorLogsConfigSchema — YAML field validation + defaults
